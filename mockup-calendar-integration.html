実装ガイドからの変更点

07_カレンダー統合.md（初期実装ガイド） → モックアップでの変更と経緯

1. 一覧画面: 店舗選択が前提 → 全店舗を常時一覧表示

実装ガイドsearchParams.shopId で店舗選択後にルール表示
モックアップ全店舗のルールをデフォルト一覧表示。店舗フィルタなし
経緯: 実装ガイドは既存UIの「店舗選択→表示」を踏襲していたが、統合画面では全店舗を一覧で見渡せる方が運用効率が良いと判断。

2. 一覧画面: 種別バッジ付き行 → 1ルール1行 + 店舗フラグチップ

実装ガイド月額ルール/時間利用ルールを種別バッジ（_typeLabel）で別行表示
モックアップ1ルール=1行。店舗名の下に有効種別をチップ表示（種別カラム不要）
経緯: 統合カレンダーの目的は「3種類を1画面で管理すること」。種別ごとに別行にするとむしろ情報が分散する。

3. 一覧画面: 直近1週間の営業予定カラム（新規追加）

実装ガイド記載なし
モックアップ適用期間の右に7日分の営業予定を色分け表示
経緯: 一覧画面だけでは「どの店舗が今週どう営業しているか」がわからない。編集画面を開かずに営業状況を一目で確認できるようにした。

4. 休業日: 単一チェック → 種別ごとに独立

実装ガイドclosed 1つ（月額の休業 = 全種別休業）
モックアップ月額・時間利用・見学それぞれに独立した休業チェック
経緯: 実装ガイドの問題点 #1。「月額は休業だが時間利用は営業」というケースが実運用で発生しうる。月額のclosed一本では対応できない。

5. 基本設定テーブル: 4列 → 8列

実装ガイド曜日 / 月額時間 / 時間利用時間 / 見学時間 / 休業（1つ）の4列
モックアップ曜日 / 月額（時間+休業）/ 時間利用（時間+休業）/ 見学（時間+不可+不明）の8列
経緯: 休業の独立化（#4）に伴い、各種別に個別の休業/不可チェックが必要に。見学は◎/△/×の3状態があるため「不可」「不明」の2チェック構成。

6. 祝日設定行（新規追加）

実装ガイド日〜土の7曜日のみ
モックアップ日〜土 + 祝日の8行
経緯: 実装ガイドでは祝日の扱いが未定義。実運用では祝日は平日と異なる営業時間にすることが多く、独立した設定行が必要。

7. 特定日カスタマイズ機能（新規追加）

実装ガイド記載なし（ShopBusinessHourException の存在は暗黙）
モックアップカレンダー日付クリック → ダイアログで種別ごと変更。他店舗反映・基本設定に戻す（スコープ選択）
経緯: 既存DBに ShopBusinessHourException テーブルが存在するが実装ガイドでは言及なし。臨時休業や祝日対応に不可欠なため統合UIに組み込んだ。複数店舗で同じ臨時変更を適用する運用ニーズに応じて他店舗反映も追加。

8. 一括設定機能（新規追加）

実装ガイド記載なし
モックアップ曜日間コピー（平日一括/土日祝一括）、種別間コピー（月額→時間利用/見学）、既存店舗からコピー
経緯: 種別×曜日の組み合わせで設定項目が大幅に増加（8行×3種別=24設定）。手動入力では運用負荷が高すぎるため、効率的な入力支援を追加。

9. 新規追加: パラメータ遷移 → Step形式

実装ガイド?shopId=...&unified=1 で編集画面に遷移するのみ
モックアップStep1（店舗・期間）→ Step2（曜日別設定）→ Step3（見学締切）→ 確認サマリー
経緯: 実装ガイドでは unified=1 パラメータの処理が未定義（問題点 #4）。設定項目が多いため段階的なフローの方がわかりやすく、確認サマリーで保存前に内容を確認できる。

10. 実装ガイドで未定義だった項目（6件）

# 問題 モックアップでの対応
1 休業日が月額の closed のみ参照 種別ごとに独立した休業設定（変更点 #4）
2 保存ロジック未定義 確認サマリーで保存内容を表示（実装時に定義要）
3 データ読み込みロジック未定義 既存店舗コピー機能で読み込みフローを提示
4 unified=1 パラメータ未定義 Step形式の新規追加フローに変更（変更点 #9）
5 getBusinessHourForDate() 未定義 カレンダー表示で曜日ベースの描画ロジックをモック化
6 店舗フラグ（canApplyMonthly 等）の実在未確認 DB調査済み。Shop.canReservePreview は実在。モック上でフラグチップとして表示
